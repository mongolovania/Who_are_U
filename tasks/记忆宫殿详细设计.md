### 架构边界：Memory Palace ≠ 用户画像引擎（2026-06-13 确立）

Memory Palace 的职责是**记忆的存储·检索·推理**——不承担用户画像（Persona/Archetype）的计算与维护。系统边界如下：

```
┌─────────────────────────────────────────────────────────────┐
│  用户画像系统 (Flutter/Dart·独立演进)                        │
│  10题测试 → 3特质 → 24原型 → 大师选择                        │
│  ⚠️ 静态快照 → 需升级为动态画像                               │
│  改进路径：PersonaTrait.score 随时间/情绪/对话演化            │
│           接入 MP API 信号(valence·脆弱性·脚本偏离)           │
│           DDI 感知置信度标注                                  │
├─────────────────────────────────────────────────────────────┤
│                     │  API 调用（只读 MP 信号）              │
│                     ▼ 不修改 MP 内部逻辑                     │
├─────────────────────────────────────────────────────────────┤
│  Memory Palace v0.9.0 (Python·不动)                            │
│  职责：记忆存储·检索·推理                                     │
│  六维跑分均第一：简单回忆·多跳推理·时间推理·情感记忆·         │
│                 因果推理·跨引用                              │
│  862 tests green · Benchmark #1 47/75                        │
│  ❌ 不承担画像职责                                           │
│  ✅ 对外提供信号 API: valence/arousal·脆弱性指数·脚本偏离标记  │
└─────────────────────────────────────────────────────────────┘
```

**原则**：Memory Palace 在六个认知维度上均已达理论最优（详见 [Memory Palace 计划论证](../code/Memory%20Palace/plans/MemoryPalace_plan.md) §8-11）。**禁止修改 MP 内部检索/存储/推理逻辑以适配画像需求**——这是单向依赖：画像系统依赖 MP 信号，MP 不对画像系统做任何假设。

**MP 为画像系统提供的信号（已有·不新增模块）：**

| 信号 | 来源模块 | API | 画像用途 |
|------|---------|-----|---------|
| valence / arousal | Retrieval Engine (emotion path) | `/api/memory/emotion_state` | PersonaTrait.score 动态更新 |
| 脆弱性指数 | vulnerability_model | orchestrator 返回 | 存储/响应阈值调节 |
| 脚本偏离标记 | script_deviation | orchestrator 返回 | 重要性信号·决策触发 |
| DDI 指数 | dda_controller | orchestrator 返回 | 原型置信度标注 |
| 记忆重要性分数 | importance_fusion | orchestrator 返回 | trait 关联记忆权重 |

## 二、L0: DDA 自适应层

### 2.1 Data Density Index (DDI)

```python
DDI = weighted_sum([
    total_sessions * 0.20,           # 总会话数
    sessions_per_week * 0.25,         # 使用频率
    avg_session_duration * 0.15,      # 平均会话时长
    avg_session_depth * 0.15,         # 平均倾诉深度（情感强度×话题深入度）
    days_since_first_use * 0.10,     # 使用历史跨度（天数/30归一化）
    session_regularity * 0.10,        # 使用规律性（间隔标准差归一化）
    time_of_day_pattern * 0.05,       # 时间段分布（凌晨=高紧迫度加分）
])
```

### 2.2 DDI 四级策略矩阵（v0.9.0 实际实现）

```
                COLD(0-10)      WARM(10-50)        HOT(50-200)          RICH(200+)
存储门禁        存全部            LLM知识判断         统计偏离             全模型(含脆弱性)
                (最小假设)       (内容判断)          (个人基线)           (个人+脆弱性)

检索策略        cold_fusion      cold_fusion         three_way            four_way_ws
                BM25(50%)        BM25(50%)           vector+BM25+graph    三路+Working Self
                +Emotion(25%)    +Emotion(25%)       (3路径融合)           (4路径+WS重排)
                +Temporal(25%)   +Temporal(25%)

情绪策略        query_driven     query_driven        personal_baseline    fine_tuned
                (查询词匹配)     (查询词匹配)         (个人基线)           (持续演化)

脆弱性评估      不可用            不可用              启用                  全量
                (无个人基线)     (无个人基线)         (个人+混合)           (精确)

衰减速率        无衰减            全局默认λ=0.05      个人自适应λ=0.05     完全个性化
                (保护冷启动)     (通用参数)           (部分个性化)
```

**v0.9.0 关键变化**（vs 原设计）：
- **COLD/WARM 共享 cold_fusion**: 根据 BEIR/MemGPT/Mem0 文献共识，DDI < 100 时 BM25 严格优于稠密向量。COLD 和 WARM 使用相同的 BM25-dominant 检索（3-path light fusion），存储策略（LLM gate、decay）是 COLD/WARM 的区分器
- **COLD 不再是"全量返回"**: v0.9.0 修复——原 `_retrieve_all()` 完全忽略 query，被 cold_fusion 替代。cold_fusion 有 fallback 链：BM25 → fuzzy → return_all
- **WARM 不再是"语义相似+时间"**: 原 `_retrieve_semantic_time()` 使用 `fuzz.partial_ratio`（Levenshtein 编辑距离），不是真正的检索算法。v0.9.0 改为使用与 COLD 相同的 BM25-based cold_fusion
- **DDI 单调性保证**: 四级检索质量单调递增（cold_fusion ≤ cold_fusion < three_way < four_way_ws）

**关键原则（宪章级约束）：**
- **零跨用户数据流**：所有模型只能使用 (a)该用户自己的数据 OR (b)LLM训练语料中的通用知识 OR (c)公开学术研究结论
- 模型复杂度 = min(个人数据量, v9模型需求) — Vapnik 结构风险最小化
- 冷启动默认：假设每次会话重要 — 最小假设原则

### 3.4 DecayEngine — 自适应衰减

```python
class DecayEngine:
    """
    基于 Ebbinghaus 遗忘曲线的自适应衰减引擎

    核心公式:
      Score = importance × retrieval_boost × flashbulb_protection

        """计算当前分数"""

    def apply_life_tick(self):
        """每日衰减更新（Sleeptime 触发）"""

    def get_forgotten(self, threshold: float = 0.1) -> list[str]:
        """获取衰减到阈值以下的记忆（供清理）"""
```

### 4.1 Working Self — 活跃目标追踪

```python
class WorkingSelf:
    """
    基于 Conway SMS 模型的 Working Self 实现

    Working Self 决定了什么记忆被激活、什么被抑制。
    它包含用户当前活跃的 goals + concerns + self-concept。
    """

    def __init__(self, user_id: str):
        self.active_goals: list[Goal] = []
        self.concerns: list[Concern] = []
        self.self_concept: dict = {}

    def infer_from_session(
        self,
        user_message: str,
        session_meta: SessionMeta,  # 时间·时长·情感强度
        ddi_level: DDILevel,
    ):
        """
        从单次会话推断 Working Self 的变化。

        DDI 自适应:
          COLD: 仅从会话元信号推断（凌晨3点→高紧迫度）
          WARM+: 从累积记忆网络推断活跃目标
        """

    def match(self, memory: MemoryNode) -> float:
        """
        计算记忆与 Working Self 的相关度 (0-1)
        相关度高的记忆在 breath 时优先浮现
        """

    def update_after_session(self, insights: list[str]):
        """对话后更新活跃目标和关注点"""
```

### 4.2 MultiSignal Importance — 多信号融合重要性

```python
class ImportanceFusion:
    """
    7信号融合的重要性评分模型

    信号:
      1. statistical_deviation  — 统计偏离 (Schank·0-1)
      2. emotional_intensity    — 情绪强度 (Bower·0-1)
      3. emotional_meaning      — 情绪意义 (Bower·0-1·语义判断)
      4. user_explicit_mark     — 用户显式标记 (0 or 1)
      5. retrieval_frequency    — 检索频率 (Ebbinghaus·归一化)
      6. association_density    — 关联密度 (A-MEM·入度/出度)
      7. working_self_match     — Working Self匹配 (Conway·0-1)

    分层计算:
      同步路径(存储时): 信号 1·2·4·5 (低成本·无LLM)
      异步路径(后台):   信号 3·6·7 (需语义理解或网络计算)
    """

    def compute_sync(self, memory: MemoryNode, user_emotion: ValenceArousal) -> float:
        """存储时的即时重要性评分（<10ms）"""

    def compute_async(self, memory: MemoryNode, graph: MemoryGraph, ws: WorkingSelf):
        """后台深度评分（异步·可调用轻量LLM）"""

    def evolve(self, memory_id: str):
        """
        重要性随时间演化——不是一次性判定，而是涌现属性。
        关联密度增长 → 重要性上升
        检索频率增长 → 重要性上升
        Working Self 变化 → 重要性重新评估
        """
```

### 4.3 Vulnerability Model — 脆弱性调节

```python
class VulnerabilityModel:
    """
    基于四个理论的脆弱性评估

    理论来源:
      - McEwen (1998): Allostatic Load — 非稳态负荷
      - Post (1992): Kindling — 压力敏化·阈值递减
      - Kuppens (2012): Emotional Inertia — 情绪惯性
      - Scheffer (2009): Critical Slowing Down — 临界减速

    Vulnerability Index (0-1):
      综合四个理论信号 → 评估用户当前心理调节容量的剩余程度

    当 VI 高时（用户脆弱）:
      → 存储阈值降低（微小事件也可能极其重要）
      → 情绪信号权重提升
      → 检索时优先浮现支持性记忆
    """

    def compute_index(self, user_id: str) -> float:
        """计算脆弱性指数"""

    def get_allostatic_load(self) -> float:
        """非稳态负荷：慢性压力累积"""

    def get_kindling_count(self) -> int:
        """点燃历史：既往重大情绪事件次数"""

    def get_emotional_inertia(self) -> float:
        """情绪惯性：连续多日valence自相关"""

    def get_critical_slowing(self) -> float:
        """临界减速：恢复速度·自相关·方差"""

    def adjust_storage_threshold(self, base_threshold: float) -> float:
        """根据 VI 调整存储门禁阈值"""
```

## 五、L3: 推理编排层

### 5.1 MemoryOrchestrator — 同步/异步分离

```python
class MemoryOrchestrator:
    """
    同步路径（用户在线·<2s·零额外LLM调用）:
      breath → inject → LLM推理 → 回复

    异步路径（后台深度处理）:
      extract → pre-filter → hold → 关联边 → 进化 → 叙事 → 衰减 → 预计算
    """

    async def breath(
        self,
        user_id: str,
        conversation_context: dict,
        ddi_level: DDILevel,
    ) -> BreathResult:
        """
        对话前浮现记忆（同步·<500ms·零LLM）

        四路检索（DDI自适应激活）:
          COLD: 全量返回（无数据可过滤）
          WARM: 语义相似 + 时间排序
          HOT:  向量(35%)+BM25(20%)+LLM关联边(25%)+情绪匹配(15%)+WS二次排序
          RICH: HOT + EmergentProperties(A-MEM)
        """
        memories = self.retrieval_engine.search(
            query=conversation_context,
            top_k=20,
            ddi_level=ddi_level,
        )
        return BreathResult(memories=memories)

    async def hold(
        self,
        user_id: str,
        conversation_id: str,
        insight: str,
        emotion: ValenceArousal,
        memory_type: MemoryType,
    ):
        """
        对话中标记记忆（同步存储·异步深度处理）

        同步部分:
          1. 统计 pre-filter（<10ms）
          2. 写入存储

        异步部分（消息队列）:
          3. LLM关联边创建
          4. 记忆进化（触发旧记忆更新）
          5. 叙事合并
        """

    async def dream(self, user_id: str, conversation_id: str):
        """
        对话后消化（异步·Sleeptime Compute）

        1. 自省消化：总结本次对话的 feel
        2. 重复议题检测：这个话题出现几次了？
        3. 叙事合并：是否需要生成新的叙事摘要？
        4. 衰减更新：Life Tick
        5. 预计算：为下次 breath 准备
        """
```

### 7.6 v0.9.0 检索引擎修复（2026-06-10 Benchmark 驱动）

> **背景**：BM25 vs MP 基准测试揭露了五个根本缺陷。BM25 Baseline（实际为关键词重叠率 `|Q∩M|/|Q|`）在 22 条合成记忆上排名 #1（43/75），MP（修复前）排名 #7（32/75）。修复后 MP 47/75（#1），全量 862 tests green。
> 详见 [hazy-baking-puffin.md §BM25 vs MP 根因分析](../code/Memory%20Palace/plans/hazy-baking-puffin.md)。

#### Fix 1: 不可用路径权重归零·不重分配

**问题**：当 vector/graph/narrative/ppr 四个路径的 infra 不可用时（测试场景），其权重（~56%）被按比例重分配给所有剩余路径，包括 emotion/temporal/cross_ref 三个噪声路径。噪声权重被放大 2-3 倍。

**修复**（`retrieval_engine.py:911-943`）：
- 不可用路径权重直接归零
- 仅在 content-matching 路径（vector+bm25）之间做归一化
- 噪声路径（emotion/temporal/cross_ref）保持原始设计权重，不被放大

```python
# 不可用路径归零
for k, available in _available.items():
    if not available and k in weights:
        weights[k] = 0.0

# 仅在 content-matching 路径间重归一化
_content_paths = {"vector", "bm25"}
_active_content = sum(weights[k] for k in _content_paths if _available.get(k, False))
if _active_content > 0:
    for k in _content_paths:
        if _available.get(k, False) and weights[k] > 0:
            weights[k] /= _active_content
```

#### Fix 2: 路径判别力自动检测

**问题**：非判别路径（如无时间约束时 temporal_score 全为 0.5）以死重量参与融合，稀释了有效信号的区分度。

**修复**（`retrieval_engine.py:1521-1546` + `1165-1189`）：
- 新增 `_detect_discriminating_paths()` 方法
- 对每条路径计算跨候选记忆的得分方差和极差
- 方差 < 0.005 且极差 < 0.05 → 自动静默该路径权重
- 剩余判别路径重新归一化

```python
@staticmethod
def _detect_discriminating_paths(
    all_results: dict[str, dict],
    score_keys: list[str],
    variance_threshold: float = 0.005,
    range_threshold: float = 0.05,
) -> set[str]:
    """检测具有实际判别力的路径"""
    ...
```

#### Fix 3: random_surface_probability 15% → 3%

**问题**：15% 的随机探索概率意味着每 ~7 次查询就有 1 次随机注入噪声记忆。在 22 条小语料场景中直接污染结果集。

**修复**（`retrieval_engine.py:640-643`）：
- 从 0.15 降至 0.03，保留 serendipity 收益但避免小结果集污染

#### Fix 4: 测试遵守 DDI 设计契约（COLD 策略）

**问题**：测试以 RICH 策略（top_k=10，设计用于 1000+ 次会话）运行 22 条 COLD 阶段记忆，直接违反 DDI 设计规范（Vapnik 结构风险最小化——复杂模型在小数据上必然过拟合）。

**修复**（`test_algorithm_comparison.py:111-114`）：
- `DDILevel.COLD` + `top_k=25`（return_all，确保 22 条记忆全量返回）
- 这是 DDI 设计的原始意图——COLD 阶段不做过滤，最小假设原则

#### Fix 5: 两阶段检索-排序解耦

**问题**：单阶段 8-path 融合让噪声路径（emotion/temporal/cross_ref）不仅能重排相关记忆，还能把**无关记忆拉进 final top-k**——噪声路径充当了召回角色而非纯排序角色。

**修复**（`retrieval_engine.py:1191-1251`）：
- **Phase 1（召回）**：仅用 content-matching 路径（bm25+vector）排序，截断至 top_k × 3
- **Phase 2（重排）**：在缩小的相关候选集上做全 8-path 融合
- 噪声路径只能在已召回的相关记忆内部重新排序，不能引入无关记忆

```
原架构：query → [8-path 单阶段融合] → top_k 结果
                                ↑ 噪声路径可拉入无关记忆

新架构：query → [bm25+vector 召回] → top_k×3 候选
                  → [8-path 全融合重排] → top_k 结果
                     ↑ 噪声路径仅重排·不召回
```

| 指标 | 修复前 | 修复后 | Δ |
|------|--------|--------|-----|
| MP 总分 | 32/75 | 47/75 | +15 |
| MP 平均分 | 1.28 | 1.88 | +0.60 |
| MP 排名 | 7/8 | 1/8 | +6 |
| 多跳推理 | 1.00 | 2.40 | +1.40 |
| 因果推理 | 1.33 | 2.33 | +1.00 |
| 情感记忆 | 1.00 | 1.75 | +0.75 |

> **本质判定**：Fix 1/2/3/5 是模型机制变更（权重归一化逻辑·新增门控·超参调整·架构重构），Fix 4 是测试方法修正（用对 DDI 策略）。4/5 修复了模型真实缺陷，1/5 修正了测试对模型的使用方式。

### 12.2 causal_verifier — 因果边验证 (P1)

**理论**：Pearl 因果之梯第 1 层（Association）→ 第 2 层（Intervention）的桥梁。
causal edges 基于关键词共现创建（`_build_typed_edges`），causal_verifier（v0.7.0 已实现）负责排除伪相关。

```python
class CausalVerifier:
    """验证记忆图中因果边的有效性，排除伪因果关系。"""

    def verify_edge(self, edge_id: str, graph: MemoryGraph) -> CausalVerificationResult:
        """
        验证单条因果边：
          1. 时间优先性：因必须先于果
          2. 一致性：因与果之间共享实体/概念
          3. 非循环性：不能形成闭环因果
        返回：验证结果 + 调整后的置信度
        """

    def verify_subgraph(self, root_id: str, graph: MemoryGraph, depth: int = 3) -> list:
        """验证从根节点可达的所有因果边，标记可疑边"""

    def verify_all(self) -> dict:
        """批量扫描所有因果边，返回验证报告"""
```

**数据模型** (`CausalVerificationResult`):
- `edge_id`: 被验证的边 ID
- `valid`: 是否通过验证
- `confidence`: 调整后的置信度 (0-1)
- `issues`: 发现的问题列表（时间乱序/概念不匹配/循环因果等）
- `verified_at`: 验证时间戳

**集成点**：
- `_build_typed_edges()` 创建因果边后立即调用 `verify_edges_for_node(new_memory_id)`
- `_stage_consolidate()` 睡眠周期的 CONSOLIDATE 阶段调用 `verify_all()`

### 12.3 Narrative → Causal Bridge (P1)

**理论**：Schank (1990) — 故事隐式地编码了因果结构。当 Narrative Engine 在叙事摘要中检测到因果模式（"因为A...所以B..."），自动创建因果边。

**修改模块**：`narrative_engine.py` 新增方法：

```python
def extract_causal_edges(self, thread_id: str, memory_graph: MemoryGraph) -> int:
    """
    从叙事线程中提取因果关系并创建因果边。

    检测中文因果模式：因为...所以... | 导致 | 引起 | 造成 | 使得 | 于是 | 因此 | 由于
    对线程中时间相邻的 moment (A→B) 检查 content_summary 是否匹配因果模式。
    如果匹配 → memory_graph.add_edge(causal, properties={"source": "narrative_bridge"})
    返回：创建的边数量。
    """
```

**集成点**：`sleeptime_compute._stage_consolidate()` 在 narrative merge 后调用。

### 12.7 memory_load_monitor — 自适应睡眠触发 (P2)

**理论**：McClelland et al. (1995) 互补学习系统（海马体→皮层记忆迁移），Diekelmann & Born (2010) 睡眠依赖性记忆巩固（*Nature Reviews Neuroscience*）。

```python
class MemoryLoadMonitor:
    """
    自适应睡眠周期触发——替代固定间隔触发。

    指标：
      - new_memories_since_last_sleep: 自上次睡眠以来的新增记忆数
      - avg_importance: 新增记忆的平均重要性
      - edge_density: 图边/节点比（高密度 = 需要巩固）
      - emotional_volatility: 近期valence/arousal方差（高波动 = 需要睡眠）
      - time_since_last_sleep: 距上次睡眠的小时数
    """

    def compute_load(self, user_id: str, bucket_mgr, graph: MemoryGraph) -> MemoryLoad:
        """计算当前记忆负载"""

    def recommend_sleep_cycle(self, load: MemoryLoad) -> dict:
        """
        返回：
          - should_sleep: bool
          - urgency: 0-1
          - recommended_stages: [REPLAY, PRUNE, ...] 子集
          - recommended_intensity: light | normal | deep
        """

    def record_sleep_complete(self, result: dict):
        """记录睡眠完成，更新内部状态"""
```
