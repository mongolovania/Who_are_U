## 一、产品定位与交互模型

### 1.1 一句话定位

**一个住在异世界山脉里的陪伴角色，既能日常闲聊陪伴，也能在你需要时提供30+位大师方法论驱动的决策支持。**

### 1.2 双价值交付

| 价值维度 | 用户感知 | 技术支撑 |
|---------|---------|---------|
| **情感价值** | 我之山成长可视化 · 独影剪影 · 里程碑 · 外观装扮 · 称号 · 温暖陪伴 | 陪伴对话引擎 + 记忆浮现 |
| **实用价值** | 决策引导 · 大师方法论 · 决策报告 · 后续追踪 | 24原型 + 50大师蒸馏 + 5阶段对话 |

两条价值线共用同一陪伴角色、同一对话界面、同一记忆底座。用户在一次对话中可能从日常闲聊自然过渡到深度决策。

### 1.3 交互架构

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│           主界面：我之山 3D 场景                        │
│       （3D可旋转·转场由远至近放大·按钮极少）              │
│                                                      │
│  🏔️ 山体                                             │
│  ├── 🌫️ 迷雾层（免费用户大部分覆盖）                    │
│  ├── 🪨 里程碑节点（点击→弹窗查看详情）                  │
│  ├── 👤 独影剪影（里程碑动作定格·随使用深度变化）         │
│  └── 🐾 陪伴角色/造物（点击→进入对话页面）              │
│                                                      │
│  [装扮入口] → 装扮+称号页面（极少按钮之一）              │
│                                                      │
│     点击陪伴角色 ──→ 转场(放大/过渡) ──→                │
│                                                      │
│  ┌─────────────────────────────────────┐             │
│  │           对话页面                    │             │
│  │  日常闲聊（主模式·超长上下文）         │             │
│  │  ├── 温暖自然的回应风格               │             │
│  │  ├── 大师方法论自然融入                │             │
│  │  ├── [决策按钮] → 决策模式            │             │
│  │  │   ├── 综合上下文决策               │             │
│  │  │   └── 5阶段引导决策                │             │
│  │  └── 用户显式提出 → 也触发决策支持     │             │
│  │  [返回] ← 转场回到我之山              │             │
│  └─────────────────────────────────────┘             │
│                                                      │
│  [装扮入口] → 装扮+称号页面（外观物品·换装·称号墙）     │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**导航原则：**
- 主界面不堆按钮，空间即导航——点击山体不同位置
- 我之山 → 对话：通过陪伴角色/造物过渡，有转场动画（缩放+聚焦）
- 对话 → 我之山：返回，有转场动画（拉远+归位）
- 决策不离开对话页面——在对话内切换模式

### 2.3 Token 成本分析

#### 模型选择

| 场景 | 模型 | 单价（输入/输出·每百万tokens） |
|------|------|------------------------------|
| 日常闲聊（默认） | DeepSeek-V3 | ¥1 / ¥2 |
| 深度决策（按需） | DeepSeek-V3 | ¥1 / ¥2 |
| 可选升级 | GPT-4o | ¥17.5 / ¥52.5 |

#### 单次对话 Token 消耗估算

| 场景 | 系统 Prompt | 大师 Context | 记忆 Breath | 对话历史 | 每轮输入 | 每轮输出 | 平均轮次 | **单次总 Token** |
|------|-----------|-------------|------------|---------|---------|---------|---------|-----------------|
| 日常闲聊（轻） | ~2K | 0 | ~1K | ~5K滚动 | ~8K | ~1K | 10轮 | ~90K |
| 日常闲聊（深） | ~2K | ~2K | ~1K | ~10K滚动 | ~15K | ~1.5K | 20轮 | ~330K |
| 决策模式 | ~3K | ~2K | ~1K | ~12K | ~18K | ~2K | 12轮 | ~240K |

#### 月度用户成本分布（DeepSeek-V3）

| 用户类型 | 月对话次数 | 闲聊/决策比例 | 月 Token 消耗 | **月模型成本** |
|---------|-----------|-------------|-------------|--------------|
| 轻度用户 | 4次闲聊 + 1次决策 | 80/20 | ~600K | **~¥0.9** |
| 中度用户 | 15次闲聊 + 3次决策 | 80/20 | ~2.4M | **~¥3.6** |
| 重度用户 | 30次闲聊 + 6次决策 | 80/20 | ~4.8M | **~¥7.2** |
| 超重度用户 | 50次闲聊 + 10次决策 | 80/20 | ~7.5M | **~¥11.3** |

> 按 DeepSeek-V3 官方定价，¥1/百万输入 + ¥2/百万输出，加权混合单价约 ¥1.5/百万 tokens。

#### 毛利率分析

| 场景 | 月模型成本 | 月付收入 | 毛利率 |
|------|-----------|---------|--------|
| 轻度用户 | ~¥0.9 | ¥35 | **97%** |
| 中度用户 | ~¥3.6 | ¥35 | **90%** |
| 重度用户 | ~¥7.2 | ¥35 | **79%** |
| 超重度用户 | ~¥11.3 | ¥35 | **68%** |

> 综合毛利率预计 **85-95%**（以DeepSeek-V3为主模型）。年付用户因折扣，毛利率略低约5个百分点。

#### LLM 路由策略（cost-aware-llm-pipeline 优化）

基于任务复杂度自动选择模型，在保证质量的前提下最大化成本效率：

| 任务类型 | 路由策略 | 预估占比 | 节省 |
|---------|---------|---------|------|
| 意图检测·情感分类 | DeepSeek-V3 轻量模式 (max_tokens=256) | ~15% token | 70% vs 全量 |
| 日常闲聊（轻） | DeepSeek-V3 (max_tokens=1024) | ~50% token | 基线 |
| 日常闲聊（深·含记忆浮现） | DeepSeek-V3 (max_tokens=2048) | ~25% token | — |
| 决策模式（大师注入） | DeepSeek-V3 (max_tokens=4096·temp=0.5) | ~10% token | — |
| 用户主动切换 | GPT-4o | — | 溢价可选 |

#### Prompt 缓存策略

系统 Prompt 分为静态层和动态层，静态部分由 LLM 提供商侧缓存，动态部分每请求注入：

```
System Prompt 结构（~3K tokens）:
  ├── [缓存层 ~2.2K] 静态角色设定 + 陪伴角色人格 + 对话原则 + 安全边界
  └── [动态层 ~0.8K]  用户原型信息 + breath记忆上下文 + 活跃大师列表 + 当前阶段
```

> 采用 DeepSeek Context Caching：静态 Prompt 部分每次对话仅首次请求消耗输入 token，后续轮次命中缓存 → 预计节省输入 token 成本 **30-50%**。

## 六、Layer 3：记忆认知层 — Memory Palace v0.9.0

详见 [记忆宫殿详细设计](记忆宫殿详细设计.md)

### 6.1 定位

Memory Palace 是整个系统的**统一记忆底座**，不为不同功能建独立的记忆系统。闲聊、决策、里程碑、情感坐标——所有内容都存储在同一 Memory Palace 中，按内容类型做内部策略差异化。

### 6.2 核心能力

**原则**：Memory Palace 六维跑分均第一（Benchmark #1 47/75），是记忆存储与检索的权威引擎。**禁止修改 MP 内部检索/存储/推理逻辑**。画像系统的所有改进均在 Flutter/Dart 侧完成，通过 MP 已有 API 获取信号（valence/arousal·脆弱性指数·脚本偏离标记），不改 MP 一行代码。

#### 理论批判：静态快照 vs 动态画像

基于 Memory Palace 计划中 17 位专家 6 轮辩论的理论框架（Conway SMS·Bower 情绪关联网络·McEwen 非稳态负荷·Post 点燃假说·Kuppens 情绪惯性·Scheffer 临界减速），当前 6 trait → 24 原型映射存在以下理论差距：

| 理论维度 | 当前系统 | Memory Palace 计划要求 | 缓解路径 |
|---------|---------|----------------------|---------|
| **自我模型** (Conway) | 静态标签·一次性赋值 | 动态 Working Self·持续演化 | PersonaTrait.score 随时间/情绪/对话更新 |
| **情感维度** (Bower+Russell) | 1 个双极维度 (empath↔perfectionist) | 2 连续维度 (valence×arousal) + 4层情感记忆 | 接入 MP valence/arousal 信号 |
| **脆弱性感知** (McEwen+Post+Kuppens+Scheffer) | 无 | 4 理论嵌套·脆弱状态下降低阈值 | 调用 MP VulnerabilityModel API |
| **脚本偏离** (Schank) | 无基线 | 需个人脚本才能检测偏离 | 接入 MP ScriptDeviation 标记 |
| **数据效率** (Vapnik+Adomavicius) | 10题→24路分类·无DDI | COLD 阶段最小假设·DDI 感知 | DDI<10 时标记原型为「初步估计」 |
| **时间动态** (Ebbinghaus+A-MEM) | 完全静态 | 重要性随时间涌现·检索巩固 | trait.score 随对话次数衰减/强化 |

> **核心认知**：10题测试 = **快照**（snapshot），不是**画像**（portrait）。快照在 t=0 时有信息量——提供初始方向、大师选择、对话语气设定。画像需要时间——通过 Working Self 持续更新、通过脆弱性感知调节、通过情绪一致性增强。**快照→画像的演化路径已由 Memory Palace 计划定义，画像系统需遵循此路径演进。**

#### 画像演化路径（v1 不增加 Sprint 范围）

三步渐进演化，全部在 Flutter/Dart 侧完成，不修改 MP：

```
Phase 1（Sprint 3 可启动）：动态分数
  PersonaTrait.score(t+1) = score(t) × (1 - α) + 新对话情绪信号 × α
  α = f(DDI)·COLD=0.3·HOT=0.05
  → 画像随对话漂移，但原型标签不变

Phase 2（Sprint 4-5）：MP 信号接入
  接入 MP API: valence/arousal·脆弱性指数·脚本偏离标记
  存储阈值 = f(脆弱性指数)·脆弱状态下降低
  原型置信度 = f(DDI)·COLD→「初步估计」·HOT→「高置信度」
  → 画像不再是扁平标签

Phase 3（v0.2.0·50+会话后）：涌现原型
  从 Working Self + Narrative Threads + Procedural Memory 涌现原型
  测试快照权重逐步降低·行为涌现权重逐步升高
  → 原型不再是「测试出来的」·而是「活出来的」
```

- **Vapnik (1998).** *Statistical Learning Theory*. Wiley.

```
  - userMentionedMasters: [String]   ← 对话中用户提到的大师

Process:
  1. Domain → 领域大师池（12-13位）
  2. Archetype → 大师匹配度评分（24×50匹配矩阵）
  3. User injection: 用户提过的名字 → 权重×2，强制入选
  4. 去重 + 排序 → top 7

Output:
  - activeMasters: [Master]（7位参谋）
  → 对话中自然融入，决策模式时标注引用
```

#### 注入格式（决策模式时）

```
## 可参考的视角

以下是这次对话的 7 位参谋。在帮你梳理选项时，
我会自然地提到他们的方法——当然，最终选择权始终在你手中。

- 查理·芒格：逆向思维——"如果这个选择会失败，会怎么失败？"
- 邓小平：实事求是——"根据实际情况，哪个选项最可行？"
- 史蒂夫·乔布斯：直觉检验——"如果钱和恐惧都不是问题，你内心选什么？"
...

## 使用规则
- 每次只引用一位大师的视角，不堆砌
- 可以交叉验证：两位大师对同一选项得出相同/不同结论吗？
- 你可以说"换一批大师"或"我想听听XX怎么说"来调整
```

### 7.3 5 阶段引导对话

| 阶段 | 名称 | 目标 | 大师注入 |
|------|------|------|---------|
| 1 | 共情倾听 | 理解用户处境·建立信任 | ❌ 无 |
| 2 | 自我扫描 | 引导向内看·发现隐藏假设 | ❌ 无 |
| 3 | 选项拆解 | 分析各选项利弊 | ✅ 7位大师视角 |
| 4 | 内在确认 | 价值观澄清·直觉检验 | ✅ 大师交叉验证 |
| 5 | 决策输出 | 生成报告·后续追踪 | ✅ 大师引用标注 |

注入时机设计：阶段 1-2 不注入大师——先建立信任、充分了解用户。阶段 3-4 引入大师视角作为参谋。阶段 5 在报告中标注引用了哪些大师的方法论。

### 7.4 闲聊中的自然决策支持

即使用户没有点击[决策按钮]，在闲聊中如果检测到决策意图（如"我在想要不要..."、"你觉得我应该..."），系统也自然引入决策支持——但不切换为5阶段模式，而是以轻松的方式融入1-2位最匹配的大师视角。

**核心元素：**

| 里程碑节点 | 每次决策/重要对话自动生成节点，精确定位在山体坐标 | 节点坐标算法·点击检测 |
| 独影剪影 | 陪伴角色的剪影，随使用深度变化状态，里程碑时动作定格 | 剪影动画·状态机 |
| 陪伴角色/造物 | 可交互角色，点击进入对话 | 可点击区域·转场触发 |
| 旋转/缩放 | 3D空间自由旋转 + 缩放查看细节 | gesture detector |
| 转场动画 | 点击节点/角色 → 由远至近放大过渡 | 转场动画引擎 |

**3D 渲染方案选型（Sprint 2末启动 Spike → Sprint 3 Week 1 前决策）：**

> **架构决策（frontend-patterns 审查）**：3D渲染方案是项目最大技术不确定性之一，决策时机从 Sprint 4 前提早至 Sprint 2 末启动 Spike，给 Sprint 3-4 留足实现时间。

**Spike 任务（Sprint 2 Week 2 · 2天）：**
1. CustomPainter 伪3D 原型：渲染山体+迷雾+1个节点，测60fps可行性
2. Flutter Scene (impeller) 原型：加载简单glTF山体模型，测iOS/Android兼容性
3. 输出对比报告：性能·开发效率·包体积·学习曲线

| 方案 | 优势 | 劣势 | 适用 | MVP推荐度 |
|------|------|------|------|----------|

| Unity 嵌入 | 完整3D能力 | 包体积大(+30MB)·集成复杂 | 高性能需求 | ⭐ |
